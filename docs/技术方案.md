# AI 小说转剧本工具 技术方案

## 1. 技术方案概述

本项目建议采用 `前后端分离的轻量架构`，以后端 Python 服务作为核心业务中枢，负责模型调用、AI 返回 JSON 的 Schema 校验、YAML 序列化与接口输出；前端负责输入、参数配置、结果预览与交互编辑。

该方案的目标不是追求复杂架构，而是优先满足比赛对以下三点的要求：

1. 功能完整，能够稳定跑通从小说输入到 YAML 导出的闭环。
2. 架构清晰，便于展示“业务逻辑”和“工程实现”的对应关系。
3. 易于快速迭代，适合通过多个小 PR 持续提交。

## 2. 总体架构

推荐采用以下结构：

```text
frontend (React)
    |
    | HTTP API
    v
backend (FastAPI)
    |- LLM 调用模块
    |- Schema 校验模块
    |- YAML 序列化模块
    |- 项目/结果存储模块
    |
    |- SQLite
    |- YAML 文件输出
```

架构特点：

1. 前后端职责清晰，便于开发和答辩说明。
2. 核心复杂度集中在后端，符合本题“文本处理 + AI 生成”的本质。
3. 使用 SQLite 和文件存储即可支撑比赛阶段需求，不会引入额外部署负担。

## 3. 技术选型建议

### 3.1 前端

推荐：

- `React`
- `Vite`
- `TypeScript`
- `Ant Design` 或 `shadcn/ui`

选择原因：

1. React 生态成熟，适合快速搭建交互式工具界面。
2. Vite 启动快、配置轻，适合比赛项目。
3. TypeScript 能提升接口联调和数据结构约束能力。
4. UI 组件库可以显著降低表单、表格、抽屉、标签页等开发成本。

前端适合承载的功能：

1. 小说文本输入与章节检查。
2. 改编模式选择。
3. 剧本结构化预览。
4. YAML 原文查看与复制导出。
5. 单场景重生成交互。

### 3.2 后端

推荐：

- `Python`
- `FastAPI`
- `Pydantic`
- `PyYAML`

选择原因：

1. 本题核心是 AI 结构化生成、Schema 校验和 YAML 序列化，Python 开发效率最高。
2. FastAPI 的接口定义清晰，自带文档，便于调试和展示。
3. Pydantic 非常适合定义 YAML Schema 对应的数据模型。
4. PyYAML 可直接用于 YAML 导出与回读校验。

后端适合承载的功能：

1. 输入文本基础校验。
2. OpenAI-compatible 模型调用。
3. AI 返回 JSON 的 Pydantic Schema 校验。
4. JSON 到 YAML 的序列化输出。
5. 模型配置、错误处理与接口封装。

### 3.3 数据存储

推荐：

- `SQLite`
- 本地 `YAML` 文件落盘

选择原因：

1. 比赛项目不需要一开始引入 MySQL 或 PostgreSQL。
2. SQLite 零运维，便于本地演示和快速部署。
3. YAML 文件本身就是最终成果物，适合直接导出保存。

建议存储分工：

1. SQLite 存项目元数据、输入原文、生成记录、生成参数。
2. YAML 文件存最终剧本结构化输出。

## 4. 核心处理流程

推荐采用 `AI 直接生成结构化 JSON + 后端 Schema 校验 + YAML 序列化` 的流程。

### 4.1 阶段一：输入校验

输入小说原文后，后端只进行基础校验：

1. 文本不能为空。
2. 文本长度满足接口最小要求。
3. 改编模式取值合法。

作用：

1. 避免无效请求进入模型调用。
2. 保持后端职责简洁，不在代码中硬编码人名、地点、事件抽取规则。

### 4.2 阶段二：AI 结构化生成

后端将小说内容、作品标题、改编模式和 Schema 约束写入 prompt，调用 OpenAI-compatible 模型直接生成完整剧本 JSON：

1. AI 判断真实角色、地点、事件和场景边界。
2. AI 生成角色表、场景列表、动作、对白、旁白和改编说明。
3. AI 返回符合 `ScriptDocument` 结构的 JSON。

作用：

1. 避免后端规则误判人物和地点。
2. 让模型基于完整上下文统一处理剧情改编。

### 4.3 阶段三：Schema 校验与 YAML 输出

模型输出后，后端只做结构层处理：

1. 将模型返回文本解析为 JSON。
2. 使用 Pydantic 校验是否符合 `ScriptDocument`。
3. 校验通过后序列化为 YAML。
4. 校验失败时返回明确错误，不静默拼接错误结构。

作用：

1. 保证最终 YAML 可解析、可编辑。
2. 保持 AI 负责业务判断，后端负责工程可靠性。

## 5. 模块划分建议

后端建议拆成以下模块：

### 5.1 `ingestion`

职责：

1. 接收原文输入。
2. 识别章节。
3. 校验输入完整性。

### 5.2 `parser`

职责：

1. 提取角色、地点、事件。
2. 生成章节摘要和场景候选。

### 5.3 `adaptation`

职责：

1. 实现改编规则。
2. 控制叙述压缩、对白提炼、内心独白处理。
3. 根据不同改编强度调整输出策略。

### 5.4 `llm`

职责：

1. 统一封装模型调用。
2. 管理提示词模板。
3. 控制生成参数和重试逻辑。

### 5.5 `schema`

职责：

1. 定义 Pydantic 数据模型。
2. 执行结构校验。
3. 负责对象与 YAML 之间的转换。

### 5.6 `storage`

职责：

1. 保存项目数据。
2. 保存生成版本。
3. 关联输入、参数和输出文件。

## 6. 前端页面建议

首版建议控制在 4 个主要页面或区域：

### 6.1 项目创建页

功能：

1. 输入作品名。
2. 粘贴小说原文。
3. 显示章节识别结果。

### 6.2 生成配置页

功能：

1. 选择改编模式。
2. 选择输出风格或长度倾向。
3. 发起生成任务。

### 6.3 结果预览页

功能：

1. 按场景查看剧本。
2. 查看角色表。
3. 查看 YAML 原文。

### 6.4 单场编辑页或侧边抽屉

功能：

1. 修改场景内容。
2. 重新生成单场对白或动作。
3. 查看校验告警。

建议仓库按比赛要求拆分：

```text
novel2script/
  frontend/

  docs/
  samples/
```

进一步建议：

```text
backend/
  app/
    api/

frontend/
  src/
    pages/

  剧本YAML-Schema设计.md
```

这样做的好处：

1. 结构清晰，评委一眼能看懂项目组织方式。
2. 文档、代码、样例分离，便于展示。
3. 有利于后续录 demo 时按目录介绍。

## 8. 为什么不建议一开始上复杂架构

不建议首版直接使用以下方案：

1. 微服务架构。
2. 消息队列驱动的大规模异步系统。
3. 多数据库混用。
4. 重型工作流编排平台。

原因：

1. 比赛项目的核心是产品完成度和逻辑深度，不是基础设施炫技。
2. 复杂架构会显著增加联调成本和不稳定因素。
3. 当前项目的核心数据流比较线性，轻量架构更适合快速落地。

## 9. 关键工程建议

### 9.1 先定义 Schema，再写生成链路

因为本题输出不是普通聊天文本，而是结构化 YAML。先定义好 Schema，再实现生成和预览，能避免后面频繁返工。

### 9.2 所有模型输出都不要直接信任

建议始终走：

1. 模型输出文本。
2. 程序转结构对象。
3. Pydantic 校验。
4. YAML 序列化。
5. 再次解析验证。

### 9.3 保留中间结构

建议保存这些中间数据：

1. 章节切分结果。
2. 角色抽取结果。
3. 场景草案。
4. 最终 YAML。

这样做的价值：

1. 便于调试。
2. 便于答辩时展示“系统不是黑盒”。
3. 便于实现局部重生成。

## 10. 最终推荐结论

如果你要一个最稳妥、最适合比赛的技术路线，建议选择：

- 前端：`React + Vite + TypeScript`
- 后端：`Python + FastAPI + Pydantic`
- 存储：`SQLite + YAML 文件`
- 核心流程：`AI 结构化 JSON 生成 + Pydantic 校验 + YAML 序列化`

这套方案的优点是：

1. 易开发。
2. 易解释。
3. 易演示。
4. 易按小 PR 持续推进。

它不是最花哨的方案，但对当前题目来说，是最容易稳定做出成绩的一套方案。
