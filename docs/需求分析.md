# AI 小说转剧本工具 需求分析

## 1. 项目背景

许多网文作者、独立编剧和内容工作室希望将小说快速改编为剧本，用于短剧、漫画分镜、动画脚本或影视前期策划。但从长篇小说到可拍摄、可继续打磨的剧本初稿，通常需要经过章节梳理、人物识别、场景拆分、对白提炼和结构重组等多个步骤，人工成本高，且对非专业创作者门槛较高。

本项目拟开发一款 AI 辅助剧本创作工具，将 3 个章节以上的小说文本自动转换为结构化剧本，并输出为 YAML 格式，帮助作者快速得到可编辑、可继续优化的剧本初稿。

## 2. 项目目标

### 2.1 总体目标

构建一个面向小说改编场景的 AI 工具，支持用户输入多章节小说文本，自动生成结构化剧本 YAML，并提供清晰的改编结果预览与导出能力。

### 2.2 具体目标

1. 支持输入不少于 3 个章节的小说文本。
2. 自动完成章节解析、角色抽取、场景划分和剧本结构生成。
3. 以标准化 YAML 形式输出剧本结果。
4. 提供 YAML Schema 文档，明确字段定义、层级结构和设计原因。
5. 保证结果可编辑、可复用，方便后续人工润色和二次创作。

## 3. 用户分析

### 3.1 目标用户

1. 小说作者：希望快速验证作品改编潜力。
2. 编剧初学者：希望借助 AI 完成剧本初稿搭建。
3. 内容工作室：需要批量生产短剧、漫画或视频脚本雏形。
4. 教学与训练场景用户：用于小说改编、剧作课程演示与练习。

### 3.2 用户痛点

1. 小说改编为剧本需要较强的结构化能力，入门门槛高。
2. 长文本信息密集，人工拆场景和提对白效率低。
3. 改编初稿往往需要反复重写，耗时较长。
4. 通用大模型输出格式不稳定，难以直接用于后续流程。

## 4. 需求范围

### 4.1 核心范围

本项目聚焦“从小说文本到剧本初稿”的第一阶段，不覆盖完整的专业剧本排版与影视工业流程。

核心能力包括：

1. 小说文本导入。
2. 小说文本基础输入校验。
3. AI 对角色、章节、情节、地点、场景和对白进行统一结构化判断。
4. 后端将 AI 返回的结构化 JSON 校验后转换为 YAML。
5. 结果预览、编辑与导出。

### 4.2 非核心范围

以下内容不作为首版必须实现内容：

1. 专业影视制片级排版规范导出。
2. 自动生成完整分镜图或视频。
3. 多人实时协作编辑。
4. 完整版权检测与查重系统。

## 5. 业务流程分析

### 5.1 目标流程

1. 用户输入小说内容，至少包含 3 个章节。
2. 系统对文本进行基础校验，确认输入不为空且具备可处理长度。
3. AI 直接判断人物、地点、事件、场景、对白、动作和旁白等剧本要素。
4. AI 按约定 Schema 返回完整结构化 JSON。
5. 后端使用 Pydantic 校验 JSON 是否符合剧本 Schema。
6. 校验通过后，系统将 JSON 序列化为 YAML，并提供可视化预览。
7. 用户对结果进行检查、微调并导出。

### 5.2 关键处理节点

1. 输入校验：确保文本内容可被模型处理。
2. AI 结构化生成：确保人物、场景、对白、动作等信息由 AI 统一判断，避免后端规则误判。
3. Schema 校验：确保 AI 返回 JSON 可被程序解析，并符合剧本 YAML Schema。
4. YAML 转换：后端只负责格式转换和错误提示，不再承担业务抽取规则。

### 5.3 改编核心逻辑

本项目不是简单地把小说内容“换个格式输出”，而是让 AI 根据小说语义直接完成结构化剧本改编。后端不再通过硬编码规则判断人名、地点、事件或对白，而是通过 prompt 和 YAML Schema 约束 AI 输出完整 JSON，再由程序校验和转换。

改编链路定义如下：

1. 用户输入小说文本。
2. AI 根据上下文判断真实角色、地点、事件、冲突和场景边界。
3. AI 将叙述、心理描写、动作和对白统一改写为剧本结构。
4. AI 返回符合 Schema 的结构化 JSON。
5. 后端校验 JSON，并转换为 YAML 输出。

换言之，系统的输出对象不是原文摘要，而是具有明确场次、角色、行动目标和对白结构的剧本初稿。

### 5.4 小说转剧本改编规则定义

为保证输出质量稳定，系统仍需要定义小说到剧本的转换准则。但这些准则主要作为 AI prompt 约束和结果校验依据，而不是由后端硬编码规则逐条实现。AI 负责综合判断，后端负责校验结构是否符合 Schema。

#### 5.4.1 场景切分规则

1. 当时间、地点、出场人物组合发生明显变化时，应切为新场景。
2. 当叙事发生明显事件转折时，即使地点未变化，也可拆为新场景。
3. 连续心理描写或背景说明不单独成场，需附着在相关场景中。
4. 单个场景应围绕一个主要戏剧目标组织，避免把多个无关事件强行塞入同一场景。

#### 5.4.2 叙述压缩规则

1. 对环境描写、世界观铺陈、重复修饰语进行压缩，只保留影响剧情理解和舞台呈现的信息。
2. 对连续性的叙述总结为简短动作描述或场景说明，不逐句搬运原文。
3. 对与主线无直接关系的枝节内容允许弱化或省略。
4. 压缩后仍需保证人物动机、事件因果和情绪转折可被理解。

#### 5.4.3 对白提炼规则

1. 原文中已有直接引语的，优先保留其核心语义，并按剧本对白格式输出。
2. 原文没有直接引语、但存在明确交流意图的，可根据上下文补写简洁对白。
3. 长段对白应拆分为多轮交互，避免单人持续输出大段说明。
4. 对白应服务于冲突推进、信息揭示或人物塑造，避免保留文学化重复表达。

#### 5.4.4 内心独白处理规则

1. 内心活动优先转写为可表现的外部动作、表情或停顿。
2. 若内心活动对剧情判断至关重要，可保留为旁白、心理旁白或场景备注。
3. 不应将大段心理描写原样写入剧本对白。
4. 当角色动机无法通过外部行为充分表达时，允许保留少量内心旁白，但应严格控制长度。

#### 5.4.5 角色归一化规则

1. 同一角色的姓名、称谓、外号应统一映射为一个标准角色标识。
2. 路人、群演、临时人物可按功能角色归类，如“店员”“侍卫甲”。
3. 角色首次出现时应尽可能附带简短身份说明，便于阅读和排戏。

#### 5.4.6 剧情保真规则

1. 关键冲突、关键人物关系、关键情节点必须保留。
2. 非关键描写可压缩，但不得改变原著主线因果。
3. 自动生成的补充对白和动作不得凭空引入与原文冲突的重要设定。

#### 5.4.7 可编辑性规则

1. 每个场景都应是相对独立的编辑单元，便于单场重写。
2. 动作、对白、旁白尽量拆分为离散条目，避免整段大文本难以修改。
3. 输出结果应保留一定原文痕迹，但不能与原文段落一一绑定，避免结构僵化。

### 5.5 创新方向定义

为了避免产品仅停留在“直线式转换工具”，首版需求中引入以下差异化能力。这些能力不只是展示层卖点，而是直接作用于改编过程和结果质量。

#### 5.5.1 改编强度控制

系统提供不同改编档位，例如“保守改编”“平衡改编”“戏剧强化”。

1. 保守改编：尽量保留原文顺序和语义，适合忠实改编。
2. 平衡改编：适度压缩叙述，增强场景感和对白密度。
3. 戏剧强化：更强调冲突、节奏和舞台表现力，适合短剧化改编。

该能力的价值在于同一小说可生成不同风格的剧本初稿，提升作品探索空间。

#### 5.5.2 角色一致性检查

系统在生成后对角色名称、身份、出场逻辑进行自动检查，例如：

1. 同一角色是否被多个别名重复创建。
2. 某角色是否在未铺垫的场景中突然出现。
3. 对白说话人是否与场景出场角色一致。

该能力可以直接提升 YAML 结果的工程可用性和可读性。

#### 5.5.3 场景节奏诊断

系统基于生成结果给出简单节奏反馈，例如：

1. 哪些场景说明过长、对白过少。
2. 哪些章节被压缩过度，可能导致剧情跳跃。
3. 哪些连续场景缺乏冲突变化。

该能力体现产品不是单次生成器，而是“辅助改编”的工作台。

#### 5.5.4 局部重生成

用户可针对单个场景重新生成，而非整篇重跑。

1. 重新生成某一场的对白。
2. 重新生成某一场的动作描述。
3. 基于不同改编强度重写单场内容。

该能力能显著降低人工迭代成本，也是相比一次性输出更实用的创新点。

## 6. 功能需求

### 6.1 文本输入模块

功能说明：

1. 支持粘贴小说全文或分章节输入。
2. 支持识别章节标题，如“第1章”“Chapter 1”等。
3. 支持最少 3 章的输入校验。
4. 支持显示输入字数、章节数和基础检查结果。

验收标准：

1. 用户能成功提交 3 章及以上文本。
2. 对空文本、超短文本、格式异常文本做出合理拦截。

### 6.2 小说解析模块

功能说明：

1. 自动切分章节内容。
2. 提取主要角色、配角、地点和关键事件。
3. 识别叙事中的时间推进与冲突转折。
4. 为剧本生成提供结构化中间数据。

验收标准：

1. 能稳定识别章节边界。
2. 能从样例文本中提取主要角色与核心事件。
3. 中间数据结构可复用于后续生成模块。

### 6.3 剧本生成模块

功能说明：

1. 按场景而非原始段落重组内容。
2. 生成场次标题、出场角色、动作描述和对白内容。
3. 由 AI 根据改编准则处理叙述、对白、内心独白和场景说明。
4. 支持保留原著关键情节，同时适度压缩冗余叙述。
5. 支持按不同改编强度输出结果。
6. 生成统一格式的 YAML 剧本。

验收标准：

1. 输出结果符合既定 YAML Schema。
2. 至少包含角色、场景、对白等核心字段。
3. 输出 YAML 可被标准解析器正确加载。
4. 对白、动作、旁白的生成符合 prompt 中定义的改编准则。
5. 同一输入在不同改编强度下应体现风格差异。

### 6.4 AI 结构化生成与校验模块

功能说明：

1. 将场景切分、叙述压缩、对白提炼、内心独白处理等改编准则写入 AI prompt。
2. 由 AI 根据小说原文直接判断人物、章节、事件、地点、场景、动作和对白。
3. 要求 AI 返回符合 `ScriptDocument` Schema 的结构化 JSON。
4. 后端只负责 Pydantic Schema 校验、错误提示和 JSON 到 YAML 的序列化。
5. 为后续局部重生成保留场景级结构字段。

验收标准：

1. AI 输出可被后端校验为合法 `ScriptDocument`。
2. 输出至少覆盖人物、场景、事件、动作、对白和旁白等核心字段。
3. 后端不通过硬编码规则抽取人名、地点、事件或对白。
4. 校验通过后可稳定转换为 YAML。

### 6.5 创新辅助模块

功能说明：

1. 提供改编强度切换能力。
2. 提供角色一致性检查。
3. 提供场景节奏诊断。
4. 提供单场局部重生成。

验收标准：

1. 用户可明确选择改编策略并看到生成差异。
2. 系统可识别至少一类角色一致性问题并给出提示。
3. 系统可输出基础节奏诊断结果。
4. 用户可只重生成单个场景而非整部作品。

### 6.6 结果预览与编辑模块

功能说明：

1. 支持 YAML 原文查看。
2. 支持结构化剧本预览。
3. 支持用户手动修改部分字段。
4. 支持导出 YAML 文件。
5. 支持按场景查看与编辑，便于局部重写。

验收标准：

1. 用户可查看完整生成结果。
2. 用户修改后内容可正确保存或导出。
3. 导出的 YAML 文件内容完整且格式有效。

### 6.7 Schema 文档模块

功能说明：

1. 提供剧本 YAML Schema 文档。
2. 说明字段定义、层级关系、字段示例。
3. 解释 Schema 设计原因，包括可读性、扩展性和工程可用性。

验收标准：

1. 文档字段定义完整。
2. 文档与系统实际输出一致。
3. 评审能够基于文档理解生成结果结构。

## 7. 非功能需求

### 7.1 易用性

1. 界面操作流程应简洁，用户无需了解 YAML 即可使用。
2. 错误提示应明确，能指出问题所在。

### 7.2 稳定性

1. 对常见输入异常具备基本容错能力。
2. 输出 YAML 需尽量保持字段稳定，避免同类结果结构漂移。

### 7.3 可维护性

1. 前后端模块职责清晰。
2. 文本解析、AI 生成、Schema 校验应尽量解耦。
3. 代码结构清晰，便于后续扩展更多导出格式或改编策略。

### 7.4 可扩展性

1. 后续可扩展为支持不同剧本风格，如短剧、动画脚本、漫画分镜脚本。
2. 后续可增加角色卡、剧情摘要、改编强度控制等功能。
3. 后续可增加更多改编规则模板，适配不同内容形态。

## 8. 数据输出需求

首版 YAML 剧本建议至少包含以下信息：

1. `meta`：作品标题、作者、生成时间、章节范围等元信息。
2. `characters`：角色列表、角色简介、角色关系。
3. `scenes`：场次列表。
4. `scene` 内字段：场次编号、时间、地点、出场角色、场景目标。
5. `beats` 或 `dialogues`：动作、旁白、对白等内容单元。
6. `adaptation_notes`：可选，记录本场压缩策略、补写对白说明或诊断提示。

设计原则：

1. 既适合人阅读，也便于程序解析。
2. 层级不要过深，降低编辑成本。
3. 字段命名统一，便于前后端对接与校验。

## 9. 约束与风险

### 9.1 业务约束

1. 必须支持 3 章以上小说转换。
2. 必须输出 YAML，而非仅输出自然语言文本。
3. 必须额外提供 YAML Schema 文档。
4. 必须明确小说到剧本的转换规则，并能在系统行为中体现。

### 9.2 技术风险

1. 长文本超出模型上下文限制，可能影响生成完整性。
2. 角色名称别称较多时，角色识别可能不稳定。
3. 模型可能输出非标准 YAML，需要增加后处理或校验。
4. 章节节奏差异较大时，场景切分效果可能不一致。
5. 自动补写对白可能偏离原著语气或引入信息幻觉。

### 9.3 应对策略

1. 使用分章节预处理与分段生成策略，降低长文本压力。
2. 增加角色归一化逻辑，合并别名和称谓。
3. 增加 YAML 校验与自动修复机制。
4. 通过提示词约束和中间结构设计提升输出稳定性。
5. 对补写内容增加“来源类型”标记，区分原文提炼与模型补全。

## 10. 版本规划建议

### 10.1 第一阶段

1. 完成输入、解析、生成、导出最小闭环。
2. 定义 YAML Schema 初版。
3. 明确改编规则初版并映射到系统流程。
4. 完成基础 README 和演示链路。

### 10.2 第二阶段

1. 增加结构化预览界面。
2. 增加角色一致性与场景完整性检查。
3. 增加改编强度控制。
4. 优化提示词和生成质量。

### 10.3 第三阶段

1. 增加一键再生成局部场景能力。
2. 增加场景节奏诊断与更多风格模板。
3. 增加更多导出格式支持。

## 11. 验收标准汇总

项目可视为达到首版目标，应满足以下条件：

1. 用户可输入 3 章及以上小说文本。
2. 系统可自动生成结构化剧本 YAML。
3. YAML 输出符合统一 Schema，且可解析。
4. 系统已明确并实现小说到剧本的基础改编规则。
5. 系统提供至少一项可演示的创新能力，如改编强度控制或局部重生成。
6. 系统提供结果预览与导出能力。
7. 项目包含一份完整的 YAML Schema 说明文档。
8. README、演示视频与代码仓库结构满足比赛提交要求。

## 12. 结论

本项目的核心价值不只是“把小说转成 YAML”，而是建立一套可解释、可复用、可迭代的小说改编规则体系，并把这套规则产品化为一个可交互的 AI 辅助创作工具。首版应同时保证三件事：一是流程闭环；二是改编规则明确；三是有可演示的差异化能力。这样后续开发才有依据，评审时也能清楚看出项目的业务深度与产品亮点。
