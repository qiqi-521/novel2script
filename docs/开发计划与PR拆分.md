# AI 小说转剧本工具 开发计划与 PR 拆分

## 1. 文档目的

本文档用于指导项目在比赛周期内按“小步快跑、持续交付”的方式推进开发，确保满足以下要求：

1. 每个 PR 只做一件事。
2. PR 描述清晰完整，可被评审快速理解。
3. 主分支始终保持可运行或至少可继续开发的状态。
4. 避免最后一天一次性提交，形成连续 commit 和 PR 记录。

## 2. 开发原则

### 2.1 总体原则

1. 先文档、后结构、再功能、最后优化。
2. 先打通最小闭环，再做创新能力。
3. 先保证 Schema 稳定，再扩展交互和局部重生成。
4. 每个 PR 保持范围单一、变更可验证、描述可复用到答辩。

### 2.2 PR 粒度原则

每个 PR 应满足以下标准：

1. 只解决一个明确问题。
2. 代码量可控，评审能在较短时间内看懂。
3. 合并后不会破坏当前演示路径。
4. 具备最小验证方式。

不建议的 PR：

1. 同时改前端、后端、文档且无统一主题。
2. 一次性引入完整前后端全部代码。
3. PR 标题过于模糊，如“更新项目”“修复若干问题”。

## 3. 建议开发阶段

建议将项目拆为 5 个阶段推进。

### 3.1 阶段一：项目定义与基础骨架

目标：

1. 明确需求、Schema 和技术方案。
2. 建立仓库基础目录。
3. 准备后续持续开发的基础环境。

本阶段交付物：

1. 需求分析文档。
2. YAML Schema 设计文档。
3. 技术方案文档。
4. 仓库目录结构。

### 3.2 阶段二：后端最小闭环

目标：

1. 建立 FastAPI 服务骨架。
2. 建立 YAML Schema 的数据模型。
3. 实现基础输入、生成和导出接口。

本阶段交付物：

1. 后端接口可运行。
2. 能接收小说文本并返回模拟或初版 YAML。
3. 能完成最基础的 YAML 校验。

### 3.3 阶段三：前端最小闭环

目标：

1. 建立 React 页面骨架。
2. 打通文本输入、发起生成、查看结果的闭环。
3. 初步具备演示价值。

本阶段交付物：

1. 输入页。
2. 结果预览页。
3. YAML 原文查看能力。

### 3.4 阶段四：业务能力与创新能力

目标：

1. 完成改编规则落地。
2. 增加改编强度控制。
3. 增加角色一致性检查、节奏诊断或局部重生成中的至少一项。

本阶段交付物：

1. 改编规则可解释。
2. 创新点可演示。
3. 输出质量明显优于简单直出。

### 3.5 阶段五：打磨与提交材料

目标：

1. 完善 README。
2. 准备 demo 视频。
3. 补测试、样例和演示素材。

本阶段交付物：

1. README 完整。
2. 样例输入输出齐全。
3. demo 录制链路稳定。

## 4. 推荐 PR 拆分方案

以下方案适合作为比赛期内的连续 PR 节奏。每个 PR 都可单独成立，且能较好满足“每个 PR 只做一件事”的要求。

### PR-01 文档与基础目录初始化

目标：

建立项目基本文档和目录骨架。

建议内容：

1. 新增 `docs/需求分析.md`
2. 新增 `docs/剧本YAML-Schema设计.md`
3. 新增 `docs/技术方案.md`
4. 新增基础目录结构

测试方式：

1. 检查目录结构是否完整。
2. 检查文档是否可直接阅读。

### PR-02 后端服务骨架搭建

目标：

初始化 FastAPI 项目结构。

建议内容：

1. 建立 `backend/app`
2. 增加基础路由
3. 增加健康检查接口
4. 增加运行入口和依赖文件

测试方式：

1. 本地启动服务。
2. 调用健康检查接口返回成功。

### PR-03 YAML Schema 数据模型实现

目标：

将文档中的 Schema 落成 Python 数据模型。

建议内容：

1. 使用 Pydantic 定义 `meta`、`characters`、`scenes`、`beats`
2. 增加基础字段校验
3. 增加 YAML 序列化和回读验证

测试方式：

1. 使用样例对象成功导出 YAML。
2. 回读后结构校验通过。

### PR-04 小说输入与章节切分

目标：

实现原文输入校验和章节切分。

建议内容：

1. 输入文本长度检查
2. 至少 3 章校验
3. 章节标题识别
4. 切分结果结构化输出

测试方式：

1. 提交合法 3 章文本应通过。
2. 提交不足 3 章文本应返回明确错误。

### PR-05 角色与事件提取中间层

目标：

为剧本生成建立中间结构。

建议内容：

1. 角色提取
2. 别名归一化
3. 章节事件摘要
4. 场景候选草案

测试方式：

1. 给定样例文本能输出角色和事件列表。
2. 同一角色不同称谓可合并。

### PR-06 剧本 YAML 初版生成链路

目标：

打通从输入文本到剧本 YAML 输出的后端闭环。

建议内容：

1. 接入模型生成或规则模板生成
2. 输出符合 Schema 的剧本对象
3. 生成 YAML 文件内容

测试方式：

1. 输入样例小说可输出至少一份有效 YAML。
2. 结果可被 Schema 校验通过。

### PR-07 前端页面骨架与接口联调

目标：

建立前端最小可用界面。

建议内容：

1. 输入页
2. 生成按钮
3. 结果页
4. 与后端基础接口联调

测试方式：

1. 前端可发起请求。
2. 可展示返回的生成结果。

### PR-08 结构化结果预览与 YAML 查看

目标：

让用户以可视方式查看剧本结果。

建议内容：

1. 场景列表展示
2. 角色表展示
3. YAML 原文面板

测试方式：

1. 用户可查看角色、场景和对白结构。
2. YAML 原文与结构化视图一致。

### PR-09 改编强度控制

目标：

加入首个明确创新点。

建议内容：

1. 提供 `保守改编 / 平衡改编 / 戏剧强化`
2. 将模式透传到后端生成流程
3. 在输出 `meta.adaptation_mode` 中记录

测试方式：

1. 同一输入在不同模式下输出存在可见差异。
2. 模式字段正确写入 YAML。

### PR-10 角色一致性检查或场景节奏诊断

目标：

加入第二个可演示创新点。

建议内容：

1. 角色出场与对白说话人一致性检查
2. 或场景长度、对白比例、压缩过度告警

测试方式：

1. 至少能识别一类异常并给出告警。
2. 告警结果能在前端展示。

### PR-11 单场局部重生成

目标：

增加实用型创新能力。

建议内容：

1. 选择单个场景重新生成
2. 仅更新指定场景内容
3. 保留其余场景不变

测试方式：

1. 用户可仅重生成一场。
2. 重生成后整体 YAML 仍合法。

### PR-12 README、样例与演示材料完善

目标：

完善提交材料。

建议内容：

1. 完整 README
2. 启动说明
3. 依赖说明
4. 样例输入输出
5. demo 视频链接

测试方式：

1. 按 README 可完成基本启动。
2. 样例链路可以复现演示结果。

## 5. 推荐每个 PR 的描述模板

建议统一使用如下结构：

### 标题

一句话说明本 PR 新增或修改了什么。

示例：

`feat: add chapter splitter for multi-chapter novel input`

### 描述模板

```text
## 功能描述
本 PR 实现了……

## 使用方式
可以通过……触发该功能。

## 实现思路
本次采用……方式实现，核心逻辑是……

## 测试方式
1. …
2. …
```

这样做的好处：

1. 满足比赛的 PR 规范。
2. 后续可直接复用到答辩材料或周报。

## 6. 推荐 commit 节奏

建议不要每个阶段只做一次大提交，而是保持连续小提交。

推荐节奏：

1. 一个 PR 内保持 2 到 5 个 commit。
2. commit 语义清楚，例如 `docs: define YAML schema draft`、`feat: add scene model validation`。
3. 尽量每天都有推进记录，避免时间戳集中在最后一天。

不建议：

1. `update`
2. `fix bug`
3. `final submit`

这类 commit 信息对评审没有帮助。

## 7. 开发顺序建议

如果你希望风险最低，建议按以下顺序开工：

1. 先做后端 Schema 与输入处理。
2. 再做后端生成链路。
3. 再做前端最小闭环。
4. 最后补创新功能和演示打磨。

原因：

1. 这个题目的真正核心在后端结构和改编逻辑。
2. 如果先做界面，很容易出现界面有了但核心输出不稳定的问题。

## 8. 最终建议

这次比赛最容易失分的，不是“功能少一点”，而是：

1. 提交过程不连续。
2. PR 粒度过大。
3. README 和依赖说明不清。
4. 创新点讲不出来。

因此，建议始终围绕以下目标推进：

1. 每一步都能形成独立 PR。
2. 每个 PR 都有明确功能边界和验证方式。
3. 尽早形成可演示闭环，再逐步增强质量和亮点。

只要按这个节奏推进，项目不仅更稳，也更容易在评审时体现“过程质量”和“工程组织能力”。
